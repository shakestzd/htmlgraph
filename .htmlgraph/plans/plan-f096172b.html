Problem

Plan review currently depends on Marimo, a Python notebook framework, introducing a full toolchain dependency chain: uv on PATH → temp dir extraction of 7 .py files → uvx marimo run --sandbox resolving marimo>=0.22.4, pyyaml, anywidget, traitlets → Python interpreter → runtime CDN fetches for d3@7 and dagre-d3. This results in a 3-8 second cold start, network requirements for both package resolution and CDN assets, and 1,712 lines of embedded Python business logic that must be extracted, written to disk, and executed as a subprocess on every invocation. The dependency is disproportionate: Marimo serves one screen (plan review) but mandates the entire Python toolchain across every deployment environment. Critically, 70-80% of a fully internal equivalent already exists in internal/plantmpl/ and cmd/htmlgraph/api_plans.go — the migration is closing a ~960-line gap, not starting from scratch.

Goals

  1. Single binary — htmlgraph serve opens plan review in browser, no Python/uv/CDN required
  2. Instant startup — <50ms vs current 3-8s cold start (Python + package resolution)
  3. Offline-first — vendor D3/dagre-d3 so plan review works with zero network access
  4. Smaller footprint — remove ~1,712 lines of embedded Python and ~200 lines of Go embedding infrastructure from the binary
  5. Architectural alignment — plan review becomes a route in htmlgraph serve, unified with the existing dashboard rather than a separate subprocess
  6. Dashboard integration — plan review renders inside the htmlgraph serve dashboard shell, not as a standalone page in a new tab; unified navigation, visual design, and SSE updates

Constraints

  • Chat widget is the single largest gap (~600 lines of new JS/Go): SSE endpoint, bubble renderer, streaming display, and message history must be built from scratch
  • Amendment management UI (~150 lines) does not exist yet and must be completed before Marimo can be fully removed
  • D3/dagre-d3 are currently CDN-loaded; vendoring them is required to meet the offline-first goal and adds bundle management overhead
  • Marimo notebooks in prototypes/ serve as the reference implementation and must remain accessible until the internal system reaches full feature parity
  • The migration cannot ship without the chat widget if plan review without AI assistance is considered incomplete — Phase 1 (chat) gates the Marimo removal
  • Plan HTML currently has its own CSS token namespace and sidebar layout; embedding in dashboard requires reconciling design systems or rendering via iframe
  • The dashboard embedding strategy (iframe vs HTML fragment) must be decided before any slice begins — S1 and S3 modify plan_page.gohtml, and S5 may restructure the template for embedding

#1 SSE chat endpoint + streaming bubble UI L Med Pending
What
Add Go SSE endpoint POST /api/plans/{id}/chat in cmd/htmlgraph/api_plans.go and register it in the planRouter switch in the same file. The handler reads the user message from the request body, opens a subprocess to Claude CLI (or Anthropic API via ANTHROPIC_API_KEY), streams token chunks back as text/event-stream, and persists each exchange to plan_feedback with action="chat_message". Port the session-ID tracking logic from internal/notebook/files/claude_chat.py (~423 lines) into a new package internal/planchat/backend.go. On the JS side, add a chat panel to internal/plantmpl/templates/plan_page.gohtml: input textarea, send button, scrollable bubble list (user/assistant differentiated by data-role attr), streaming text append via EventSource, auto-scroll. Load chat history via GET /api/plans/{id}/feedback on page open (filter action="chat_message" rows). Graceful degradation: hide panel when /api/plans/{id}/chat returns 503 (Claude unavailable). Extend buildFeedbackResponse() in api_plans.go to include action='chat_message' rows in GET /api/plans/{id}/feedback responses.
Why
The chat widget is the single largest missing piece blocking the Marimo removal. Everything else in the internal plan review system already has parity. Without this slice the migration cannot proceed because the review workflow depends on the AI discussion sidebar to surface amendments and unblock approvals.

Synthesis

The plan is structurally sound but has three critical issues to resolve before approval: (1) The dashboard embedding strategy (q-dashboard-embedding) must be answered BEFORE any slice begins, because S1 and S3 modify plan_page.gohtml and S5 may require restructuring it. (2) S4's file list is incomplete — add prototypes/plan_export.py and sample_plan.yaml. (3) SSE plan events are larger than stated — use a scoped endpoint per plan instead of modifying the global stream. Additionally, S3 effort should drop from M to S given graph node coloring is 80% done.
