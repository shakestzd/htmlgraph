DesignPending

Problem

HtmlGraph's core value proposition is "see the full graph of AI-assisted work" — but the graph currently only shows work items (features, bugs, spikes, tracks) and a handful of filtered sessions. The causal chain (agent spawned → session created → feature claimed → commits produced → files modified) is entirely absent from the visualization. The raw data exists in SQLite across 5 tables (git_commits: 51 rows, feature_files: 299 rows, agent_lineage_trace: 141 rows, sessions: 139 parent→child pairs, tool_calls: 1454 rows) but none of it is surfaced as graph nodes or edges. The /api/graph endpoint only queries features, tracks, and sessions (with a >5 events filter), and the D3 frontend only has color/layout mappings for 6 node types. The internal/graph DSL and algorithms also don't model commit or file node types.

Goals

  1. **Complete causal chain** — every agent spawn, session, commit, and file modification is a visible, clickable node in the graph
  2. **Provenance in one click** — clicking any node reveals its full upstream/downstream chain (who spawned the agent, what feature it worked on, what commits it produced, what files were touched)
  3. **Query API** — REST endpoints that answer 'what commits came from agent X?', 'which sessions touched file Y?', 'what agents worked on feature Z?' returning structured JSON
  4. **Performance at scale** — graph renders smoothly with 2000+ nodes; query-time derivation for cheap joins, materialization only for hot dashboard paths

Constraints

  • Hybrid materialization — derive edges at query time where joins are cheap (< 50ms); only materialize into graph_edges for edges used in hot paths (graph API, lineage command)
  • No schema-breaking changes — extend existing tables and graph_edges, don't replace them
  • D3 force layout must remain usable — new node types need distinct visual treatment to avoid clutter (smaller radius, lower opacity, type-band positioning)
  • Backward compatible API — existing /api/graph consumers must not break; new node/edge types are additive
  • File nodes are high-cardinality — 299 feature_files today, could be 1000+ at scale. Must support filtering or pagination
  • Lineage command BFS (lineage.go) only traverses graph_edges — new edge types must be materialized during reindex for CLI commands to see them (documented at lineage.go:212-215)
  • feature_files.session_id is nullable — edge derivation must handle NULL gracefully

#1 Commit and file nodes in graph API M Med Pending
What
Add commit and file nodes to the /api/graph endpoint in cmd/htmlgraph/api_graph.go. 1. Add a new query branch in loadGraphNodes (after the sessions query, ~line 250) that reads from git_commits using SELECT DISTINCT commit_hash to avoid duplicate nodes (git_commits has a composite PK of (commit_hash, session_id), so a commit may appear multiple times across sessions): SELECT DISTINCT commit_hash, 'commit', SUBSTR(message,1,80), 'done', feature_id, session_id FROM git_commits GROUP BY commit_hash. Map to graphNode with id=commit_hash, type="commit", title=truncated message. 2. Add another query branch for files from feature_files: SELECT DISTINCT file_path, 'file', file_path, '' FROM feature_files. Map to graphNode with id=file_path, type="file", title=basename(file_path). Deduplicate by file_path since multiple features may touch the same file. 3. Create edges: commit→feature (type="committed_for"), commit→session (type="produced_by"), file→session (type="produced_in") using feature_files.session_id (which is a direct FK), file→feature (type="touched_by"). Note: file→commit (modified_in) edges cannot be derived directly because feature_files has no commit_hash column. Users traverse file → session → commit in two hops instead. Handle NULL session_id in feature_files gracefully (skip edge creation when session_id IS NULL). 4. Update the graphNode struct if needed (no new fields required — id, type, title, status, edges, activity already cover everything).
Why
This is the foundational slice. Without commit and file nodes in the API response, no downstream work (frontend, provenance, queries) can surface them. The data already exists in git_commits and feature_files — this slice just queries it and formats it as graph nodes with derived edges.

Synthesis

The plan is fundamentally sound but has three critical issues that must be addressed: (1) file->commit edges cannot be derived (no FK) — replaced with file->session edges, (2) edge materialization strategy must be materialize-on-reindex for CLI compatibility, (3) git_commits composite PK requires deduplication. All three are addressed in this revision. The remaining medium-severity risks (D3 performance at scale, session filter threshold) are manageable with per-type node budgets and benchmarking.
