Plan: Migrate every SQLite consumer to HTML+NDJSON (Phase 3)

Problem

~110 Go files import `database/sql` across CLI commands, dashboard endpoints, hook handlers, and four internal packages (`graph`, `agent`, `planchat`, `registry`). Until every consumer reads from HTML or NDJSON instead of SQLite, the driver cannot be deleted. The five migrations in this sub-plan are largely independent and can land in any order; the build-tag stub introduced in slice-1 provides the compile-time guarantee that proves convergence.

Goals

  1. `go build -tags sqlite_disabled ./...` succeeds; `go tool nm` shows zero `database/sql` symbols in the binary.
  2. Every CLI command, dashboard endpoint, hook handler, and internal package reads canonical sources (HTML + NDJSON + git) instead of SQLite.
  3. Activity feed reads hook events and OTEL spans from the same NDJSON sink — they cannot diverge.

Constraints

  • Every migrated CLI command and dashboard endpoint is byte-identical or semantic-equivalent to its SQLite-backed predecessor on a representative fixture.
  • `?legacy=1` query param preserved on dashboard endpoints during this sub-plan for parity comparison; removed in Child E.
  • Build-tag stub is the binding contract — runtime flags are insufficient for transitive `sql.Open` calls.

#1 Phase 3b — Migrate SQLite-using CLI commands; introduce `sqlite_disabled` build tag stub L High Not Started Pending
What

Rewrite the SQLite-dependent CLI command read paths to use HTML + NDJSON + git directly: (1) trace: scan commits via git log + path-filter; read produced-by-session and produced-commit <link rel> edges from work-item heads. (2) lineage, lineage_tree: BFS over work-item heads using htmlparse.ParseLinks; resolve neighbor IDs via href basenames. (3) session list/show: scan .htmlgraph/sessions/*.html head metas and per-session NDJSON; sort by htmlgraph:created. (4) status: drop dbpkg.CommitAttributionRate; recompute on demand. (5) statusline: drop SQL queries; read from work-item HTML. (6) track, claim, projects: migrate to HTML+NDJSON. (7) hook, hook_exit_plan_mode, backfill_ai_title, check_cross_project: migrate the user-side hook handlers (the deeper hook-event-persistence migration is slice-15).

Introduce the build-tag enforcement: a new file internal/db/disabled_stub.go with //go:build sqlite_disabled that re-exports the internal/db API surface as functions that return a sentinel error. The matching internal/db/enabled.go with //go:build !sqlite_disabled keeps current behavior. CI builds the binary with -tags sqlite_disabled and asserts the full CLI command matrix passes; this is the binding compile-time contract that proves SQLite isn't required.

Why

These commands are the SQLite-dependent CLI paths uncovered by the database/sql import audit. Moving them off SQLite is most of the work that lets Phase 4 delete the dependency. The build-tag stub replaces the original "--block-sqlite" runtime flag (which the critic correctly noted cannot intercept transitive opens) and gives a compile-time guarantee instead.

#2 Phase 3c — `htmlgraph query` subcommand (gojq over JSON-LD) M Low Not Started Pending
What

Add htmlgraph query <jq-expr> [paths...]: extracts JSON-LD from every HTML file in the given paths (default .htmlgraph/**/*.html), pipes them through an embedded github.com/itchyny/gojq filter, and emits JSONL on stdout. Supports --type feature shorthand to scope to a single work-item type. Default output is one JSON object per file; --slurp collects into an array. Document common recipes in docs/cli/query.md (find all P1 bugs, lineage tree as JSON, etc.).

Note: cmd/htmlgraph/query.go does NOT currently exist (verified via ls cmd/htmlgraph/query*); this is a new-file slice. Critic feedback that suggested it might be partially implemented appears to have been wrong.

Why

Gives users and downstream scripts a stable shell-scriptable interface that doesn't require a new CLI subcommand for every read pattern. Replaces ad-hoc shell pipelines and prevents feature-creep pressure on the core CLI surface. Also serves as a power-user escape hatch when the dashboard doesn't expose a needed query.

Files
cmd/htmlgraph/query.go, cmd/htmlgraph/query_test.go, docs/cli/query.md, go.mod, go.sum
Done when
  • `htmlgraph query '.status' --type feature` lists the status of every feature file.
  • `htmlgraph query '. | select(.status == "in_progress")'` filters correctly.
  • Embedded gojq does not reach the network; runs offline.
  • Documented at least 5 recipes in `docs/cli/query.md`.
Test Strategy

Unit: TestQuery_TypeFilter — table-driven against schema fixtures. Unit: TestQuery_SelectExpression — assert correct subset. Integration: TestQuery_RealRepo — runs against .htmlgraph/ in CI; expected output count matches find | wc -l.

Critic Revisions
LOW researcher cmd/htmlgraph/query.go already exists. Slice may be partially implemented — read existing code before estimating effort. Could shrink to schema-aware filtering on top of the existing scaffolding.

#4 Phase 3e — Migrate hook event persistence from SQLite to NDJSON sink L High Not Started Pending
What

The internal/hooks/ package contains ~20 files that persist hook events directly to SQLite tables (e.g. agent_events, sessions). Files include: lifecycle_test.go, missing_events.go, posttooluse.go, pretooluse.go, session_end.go, session_start.go, session_sweep.go, subagent_start.go, task_tracking.go, tooluse_shared.go, track_event.go, transcript_assistant.go, transcript_user_prompt.go, user_prompt.go, yolo_guard.go.

Migrate all hook event writes to the per-session NDJSON sink (internal/otel/sink/ndjson/ndjson.go). Each hook handler emits a typed event record ({kind: "log", native: "hook_*", session_id, prompt_id, ...}) via the sink's append-only path. The dashboard activity feed (slice-10) reads the same NDJSON for both hook events AND OTEL spans — they cannot diverge.

Today's hook handlers also create rows in the sessions table. Replace with: write <meta name="htmlgraph:created"> to a session HTML file at hook session_start; subsequent updates touch the same file via the schema-aware writer (slice-5).

Why

The hook-event persistence layer is the single largest unaccounted- for SQLite consumer in the codebase. Slice-11 cannot delete SQLite while hook handlers continue writing to agent_events. By converging hook events and OTEL spans on the same NDJSON sink, the activity feed bug (the original symptom of this conversation) goes away by construction — both sources read from the same place.

#5 Phase 3f — Migrate `internal/{graph,agent,planchat,registry}` off SQLite L High Not Started Pending
What

Four internal packages depend on SQL queries beyond the read-path audit's scope: (a) internal/graph/ — algorithms.go, dsl.go, pattern.go, querybuilder.go, sessions.go. Provides the lineage/trace graph engine. Migrate to read directly from work-item HTML <link rel> edges; cache parsed graphs in-memory in a session- scoped struct. (b) internal/agent/ensure.go — agent registry persistence. Migrate to a small JSON file at .htmlgraph/agents.json (which already exists for some metadata). (c) internal/planchat/backend.go — plan feedback persistence. Plan feedback is already partially in internal/db/plan_feedback.go; migrate that store to per-plan <feedback> JSON-LD nested in the plan's HTML, or a sidecar .htmlgraph/plans/<id>.feedback.jsonl. (d) internal/registry/registry.go — host/project registry. Migrate to a JSON file at ~/.local/share/htmlgraph/registry.json (or platform-equivalent).

For each package, document the migration choice (in-memory cache vs JSON sidecar vs HTML embedding) in the package's doc.go so future readers understand WHY.

Why

Slice-11 cannot delete SQLite while these four packages still import database/sql. They were missed in the original plan because the read-path audit only enumerated CLI commands, not internal packages. This slice closes the gap.
