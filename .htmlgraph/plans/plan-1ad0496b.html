Plan: Automode support — permission-mode auto + HtmlGraph guardrails

Add 'htmlgraph auto' as a sibling subcommand to 'htmlgraph yolo' that launches Claude Code with --permission-mode auto (Anthropic's classifier-based destructive-pattern blocker, v2.1.111+) while still applying the HtmlGraph PreToolUse guard suite. Extend the bash guard to catch patterns automode catches (curl|bash, force-push), subscribe to the new PermissionDenied hook for classifier-denial telemetry, add a consecutive-deny circuit breaker so stuck-agent retry loops escalate, and document the full mode matrix.

Problem

Anthropic shipped --permission-mode auto in Claude Code v2.1.111 (2026-04-16): a classifier model pre-reviews each tool call and blocks destructive patterns (curl|bash, force-push, mass deletion, IAM grants, etc.). HtmlGraph's PreToolUse guards (work-item attribution, TDD commit, budget, UI screenshot, worktree isolation, research-first, bash-file-write blocklist) are orthogonal — domain-specific work-discipline controls automode has no concept of. Today there is no way to launch Claude with both layers active in a single command, no telemetry from the classifier, no circuit breaker for retry loops, and the HtmlGraph bash guard misses several patterns that automode catches (curl|bash, wget|sh, git push --force, tee, cp). The launchClaudeAuto() helper already exists at cmd/htmlgraph/claude.go:195 but is unreachable from any user-facing subcommand, and isYoloFromEvent() at internal/hooks/yolo_guard.go:25 only recognizes 'bypassPermissions' as a strict mode — 'auto' falls through and yolo-only guards (worktree, TDD, budget, UI, diff-review) silently disengage.

Goals

  1. **Layered defense in one command** — `htmlgraph auto` launches Claude with `--permission-mode auto` AND the full HtmlGraph guard suite (work-item, TDD, worktree, budget, UI validation, research-first), measured by: invoking the command produces both classifier blocks and HtmlGraph hook blocks for the appropriate tool patterns.
  2. **Bash guard parity with automode blocklist** — `bashFileWritePattern` (or a new sibling `bashDestructivePattern`) catches `curl|bash`, `wget|sh`, `git push --force`, `tee` to non-`.htmlgraph/` paths, and `cp` to source files; verified by table-driven tests in `pretooluse_test.go`.
  3. **Classifier-denial telemetry** — Every `PermissionDenied` hook event is recorded in `agent_events` with `event_type='permission_denied'` and the denied tool/input summary; queryable via the dashboard and `htmlgraph analytics`.
  4. **Stuck-agent escalation** — After N consecutive `PermissionDenied` events for the same session (default 3), the next PreToolUse is hard-blocked with a circuit-breaker message instructing the user to intervene; verified by integration test.
  5. **User-facing mode matrix** — A docs page (or extension to `docs/blog/posts/yolo-mode.md`) explains default vs `--permission-mode plan` vs `auto` vs `bypassPermissions` vs `htmlgraph yolo` vs `htmlgraph auto`, including which guards apply in each.

Constraints

  • Plugin source-of-truth rule — hook event matrix is declared in `packages/plugin-core/manifest.json`; `plugin/hooks/hooks.json` is generated by `htmlgraph plugin build-ports`. Never hand-edit generated files.
  • Hook state rule — gates must be reproducible across session restarts; the circuit breaker counter must read from durable state (`agent_events` SQL query), not in-memory session state.
  • Hooks must not write to stderr — debug output goes to `debug.log`. Allow returns `{}`, deny returns `{"decision":"block","reason":"..."}`.
  • `event_type` is a CHECK enum at `internal/db/schema.go:124-129` — adding `permission_denied` requires a schema migration (SQLite ALTER TABLE rebuild pattern), or we choose a different storage approach.
  • automode requires Anthropic Max+ plan — `htmlgraph auto` must fail gracefully on plans without auto-mode access (parse classifier-disabled error from claude CLI, or document as pre-flight requirement).
  • Existing 5 features (feat-2d2dfc12, feat-9c14a3ef, feat-fdad254b, feat-e86dbcdb, feat-85ef5a6b) are already filed on trk-68655b2a — slices map 1:1 to these features via the slice.feature_id field so finalize reuses them rather than creating duplicates.

What

#2 Destructive bash guard — curl|bash, wget|sh, force-push, tee, cp S Low Pending
What
Add `bashDestructivePattern` regex to `internal/hooks/pretooluse.go` (alongside `bashFileWritePattern` at line 343) that matches: `curl ... | (bash|sh)`, `wget ... | (bash|sh)`, `git push ... --force` (and `-f`), `git push --force-with-lease` to protected branches (main/master), `tee` writing outside `.htmlgraph/`, `cp` writing to repo source files, and `chmod +x` on downloaded files. Wire a new guard `checkBashDestructiveGuard` called from `PreToolUse()` BEFORE the work-item guard so destructive intent fails fast. The guard hard-blocks (returns `{"decision":"block","reason":...}`) when in strict mode (yolo or auto), and warns-only via debug.log in default mode (consistent with classifier behavior in auto mode being authoritative). Provide a deterministic backstop for users not on Max+ who can't access automode. Anchor regexes to command shape (not substrings) per the hook state rule.
Why
Goal 2 (bash guard parity). Today's bash guard catches sed/redirects/rm/mv/dd but misses pipe-from-network-to-shell and force-push — exactly the patterns automode's classifier blocks. Users on plans without automode access have zero deterministic protection from these. Even on automode, the deterministic Go guard runs first and gives a faster, citable failure (file:line) before round-tripping to the classifier.

#4 Consecutive-deny circuit breaker — escalate stuck-agent retry loops M Med Pending
What
Extend `internal/hooks/pretooluse.go` to query `agent_events` at the start of every PreToolUse: count `permission_denied` events for the current `session_id` since the last non-denied tool_call (or in the last K minutes, whichever is shorter — tunable via constant). When count >= 3 (default), hard-block the next tool call with a circuit-breaker message: `"Circuit breaker tripped: 3 consecutive permission denials in this session. Pause and review with the user before retrying."` Reset the counter when any non-denied tool_call succeeds. Add a hidden `--reset-circuit-breaker` flag to `htmlgraph` (or a session-scoped escape hatch in `permission_denied.go`) for manual unstuck. Anchor counter to durable SQL query (per hook state rule), not in-memory state.
Why
Goal 4 (escalation). Today a denied PreToolUse simply returns the block, and the agent retries indefinitely with minor variations — burning tokens, spinning forever, and obscuring the root cause. The circuit breaker forces a hard stop after a small number of denials so the user can intervene. Reading from `agent_events` (slice 3) keeps the gate reproducible across session restarts.
Files
internal/hooks/pretooluse.go, internal/hooks/circuit_breaker.go, internal/hooks/circuit_breaker_test.go, internal/db/event_repo.go, cmd/htmlgraph/auto.go
Done when
  • After 3 PermissionDenied events for the same session_id with no successful tool_call between them, the 4th PreToolUse returns `{"decision":"block","reason":"Circuit breaker ..."}`.
  • A successful tool_call resets the counter (verified by integration test: deny, deny, allow, deny, deny → not blocked).
  • The threshold is a named constant (e.g. `circuitBreakerThreshold = 3`) for easy tuning.
  • Counter query is a SQL aggregate against `agent_events`, not in-memory state.
  • Manual reset path documented (e.g. `htmlgraph session reset-circuit-breaker <session-id>` or equivalent).
  • All quality gates pass.
Test Strategy
Unit (internal/hooks/circuit_breaker_test.go): - TestCircuitBreaker_BelowThreshold — 2 denials → no block - TestCircuitBreaker_AtThreshold — 3 denials → block on 4th call - TestCircuitBreaker_ResetByAllow — deny,deny,allow,deny,deny → no block - TestCircuitBreaker_PerSession — denials in session A do not affect session B - TestCircuitBreaker_TimeWindow — denials older than K minutes do not count Unit (internal/db/event_repo_test.go): - TestCountConsecutiveDenials_Query — SQL helper returns expected counts Integration: end-to-end via fixture-driven PreToolUse calls with seeded agent_events. Regression: PreToolUse latency does not regress measurably (add a benchmark if not present).
