HtmlGraph Design Slices Decisions Critique Progress Chat

Plan: Multi-Project Hardening — per-project process isolation

Re-architect `htmlgraph serve` from shared-process query-param dispatch to per-project child-process reverse-proxy isolation. Drops the `--global` mode flag entirely: `htmlgraph serve` is the only human-facing serve command, always starts the parent with reverse proxy, shows a projects-landing doorway, and spawns per-project children on demand. When invoked from inside a project directory, the server detects the current project and prints a deep-link URL to open it directly. Single-project serve becomes an internal cobra command (hidden from help) used only by the supervisor to spawn children. OS-level process isolation; correct-by-construction hook context; crash isolation; zero SQLite handles in the parent process.

Problem

`cmd/htmlgraph/serve_global.go` implements a dual-mode `htmlgraph serve --global` path that holds a `projectCache` of read-only `*sql.DB` handles to every registered project and dispatches project-scoped routes (`/api/plans/*`, `/api/plans/<id>/chat`, `/api/plans/<id>/delete`, `/api/plans/<id>/render`, `/api/graph`, `/api/sessions`, `/api/features/*`, `/api/transcript`, `/api/events/tree`, `/api/events/subagent`) via `dispatchByProject(?project=<id>)`. The handlers then receive a read-only DB opened in the global server's working directory with the global server's env. Any write — plan chat streaming, plan finalize, plan delete, feature state transitions, hook-driven mutations — runs with the wrong CWD, wrong `HTMLGRAPH_PROJECT_DIR`, wrong `.htmlgraph/` attribution, and a SQLite handle that was explicitly opened `mode=ro`. Two concrete consequences surfaced during smoke testing on 2026-04-09: 1. `POST /api/plans/plan-368a5cd2/chat` returned HTTP 400 because the embedded plan template hardcodes `/api/plans/<id>/chat` with no `?project=` and `dispatchByProject` demands it. The band-aid fix (inject `?project=` into the template) was rejected because it papers over the symptom while leaving the architecture exposed. 2. A successfully-routed write (hypothetically) against `/api/plans/<id>/finalize?project=A` would: open a read-only DB to project A (SQLite errors on write), but first try to run hooks under the global server's CWD and env, attributing to the wrong project's session tables, writing event files to the wrong `.htmlgraph/` directory, and shelling out `claude` subprocesses under the wrong project context. The router has zero enforcement of which project a given write belongs to — the query param is a hint, not a boundary. The existing implementation also forces every frontend fetch call site to remember to append `?project=`. Today `buildProjectUrl` in `dashboard/js/helpers.js` covers 18+ call sites; the plan template's hardcoded URLs bypass it entirely. And the `--global` mode flag forces users to know which mode they want before running the command — the same tool has two behaviors depending on a flag that adds no value once the parent-proxy model ships. Both problems go away when the parent-proxy server becomes the default and single-project mode becomes an internal implementation detail.

Goals

  1. **Process isolation** — each active project runs in its own `htmlgraph serve` child process with its own OS heap, env vars, CWD, SQLite connection pool, and hook context. A memory bug, held-wrong-pointer, or router confusion cannot leak state across projects because the kernel enforces the boundary.
  2. **Global server holds zero SQLite handles** — the global process never opens any project's SQLite database, not even read-only. All database access happens inside child processes bound to their own project directory. The landing page reads only from the registry JSON file; per-project data is served by the corresponding child via `/p/<id>/api/*` proxy. This is grep-enforceable: `grep -rn 'sql.Open\|sqlx.Open\|dbpkg.Open\|registry.OpenReadOnly' cmd/htmlgraph/serve_global.go` must return zero matches.
  3. **Single user-facing origin** — the user still sees one dashboard on `http://localhost:8087/`. Drilling into a project navigates to `/p/<project-id>/` which the global server reverse-proxies to the matching child. All per-project views, plans, events, and graphs share the same origin under the `/p/<id>/` prefix.
  4. **Correct-by-construction hook context** — children are spawned with `--project-dir <path>`, resolved by `internal/paths/resolve.go` as the highest-priority source. All `.htmlgraph/` writes, hook events, and subprocess invocations automatically attribute to the right project with zero per-handler coordination.
  5. **Zero frontend ceremony** — the plan template, feature detail pages, and any future embedded view receive their project context via a template-injected `window.htmlgraphPlanBase` variable (set by the child's render path, not the global server). No `?project=` anywhere in user-facing code. `buildProjectUrl` becomes a one-line helper that branches on `window.location.pathname.startsWith('/p/')`.
  6. **Architectural acceptance test** — an integration test creates two temp projects, writes feature X to projectA through `/p/A/api/workitem/create`, and asserts projectB's SQLite file on disk does NOT contain feature X. Cross-project contamination becomes impossible at the boundary, not merely absent by inspection.

Constraints

  • Single binary, single entry point — `htmlgraph serve` spawns children of the same binary via `os.Executable()`. No new CLI tool, no separate release artifact.
  • Children bind `127.0.0.1:0` — kernel assigns a free ephemeral port. Parent reads the bound port back via a stdout handshake line before routing any requests. Must not require pre-allocation or config.
  • Crash isolation is bidirectional — a panic in child A must not affect child B, and the global server must return 502 (not hang) when a child is dead. The next request to that project respawns.
  • Idle reaping required — children older than a configurable idle timeout (default 30 min) are SIGTERMed to prevent process leaks over long dashboard sessions.
  • Zombie prevention on parent shutdown — when the global server receives SIGINT/SIGTERM, it must SIGTERM all children, wait up to 5s for graceful exit, then SIGKILL survivors. No orphan processes after `Ctrl+C`.
  • All cross-project access in the global layer is read-only. The aggregate handlers (`/api/projects/all/stats`, `/api/events/stream`, `/api/events/recent`) may open foreign project DBs in `?mode=ro` mode for aggregation but MUST NOT call any handler that performs writes. This is the only architectural rule that cannot be relaxed.
  • The `--global` flag is dropped entirely. `htmlgraph serve` is the only human-facing serve command and always starts the parent with reverse proxy. Single-project serve lives on as a hidden cobra subcommand (`_serve-child` or similar) used exclusively by the supervisor to spawn children — never typed by humans, excluded from `--help` output.
  • `active_work_items(session_id, agent_id)` attribution (feat-ab67561e) composes cleanly — each child has its own `sessions` table and its own writer queue, so there is no cross-project SQLite contention by construction. The attribution fix applies per-project without further changes.
  • Supports macOS and Linux. Windows is not a target for this work — process lifecycle differs (`pdeathsig` Linux, `ProcessMonitor` macOS, Job Objects Windows) and Windows support can be filed separately if needed.
  • Child must emit the port handshake line as its FIRST stdout output. After the handshake, child stdout and stderr must be redirected to a per-project log file at `.htmlgraph/logs/serve-<port>.log` to prevent the parent's pipe reader from blocking on subsequent prints.
  • `--project-dir` is a persistent flag on the root command, not on the `serve` subcommand. Child spawn arg order is fixed: `[binary, --project-dir, <path>, serve, --port, 0]`. Wrong order causes cobra to reject the flag at runtime.
  • SQLite WAL mode must be explicitly enabled on both the parent's read-only aggregate connections (mode=ro) and the child's read-write connection. Without WAL on both sides, concurrent readers block writers. Audited and enforced as part of slice 4.
  • Only one `htmlgraph serve` process per host. Two global servers would race on ephemeral port allocation and leave orphaned children on crash. Documented limitation; optional future work to enforce via a lockfile at `~/.htmlgraph/global-server.lock`.
  • Process isolation is at the kernel boundary, not at the access-control layer. The global server has no auth today and this work does not add any. A user with HTTP access to the global server can reach `/p/<any-registered-project>/` — the architectural guarantee is 'no cross-project contamination', not 'project-level access control'.
  • On macOS there is no `prctl(PR_SET_PDEATHSIG)` equivalent. Child processes use a 5-second heartbeat goroutine that checks the parent PID and exits if the parent is gone. Linux uses pdeathsig directly. Orphan cleanup on parent SIGKILL is best-effort.
  • Cross-project analytics are explicitly out of scope for this MVP. The landing page is a pure doorway: it lists registered projects from `~/.local/share/htmlgraph/projects.json` showing name, path, git remote, and last-seen timestamp. It does NOT show per-project work-item counts, live event streams, or aggregate statistics. These are deferred to a Phase 2 feature (tracked separately) that will design the cross-project analytics story intentionally rather than inheriting it from the MVP.
  • The global server holds ZERO SQLite connections — no `aggregateCache`, no `projectCache`, no lazy read-only reads. Every database access in `htmlgraph serve` happens inside a child process bound to a single project's directory. This invariant is verified by grep in the slice 4 done_when and by an automated test that inspects the global server's DB handle count via reflection or instrumentation.
  • When `htmlgraph serve` is invoked from inside a project directory (CWD has a `.htmlgraph/` dir), the server detects the current project by walking up to find `.htmlgraph/`, computing the project ID via SHA256-hex-8 of the absolute project dir, and checking it against the registry. If the project is registered, the startup banner prints both URLs: the doorway at `http://localhost:<port>/` AND the deep link at `http://localhost:<port>/p/<id>/`. If the project is not yet in the registry, auto-register it first (same path the current `PersistentPreRunE` uses). Detection failure is non-fatal — the server still starts and prints the doorway URL only. This is Option B from the v4 UX discussion: one server process, no mode branching, but the common case of 'I'm already in a project' gets a clickable deep link in the startup output.

#2 internal/childproc package — spawn, handshake, lifecycle, reap L Med Pending
What
New package `internal/childproc/` with `childproc.go` + `childproc_test.go`. Define `type Child struct { ID string; ProjectDir string; Port int; PID int; cmd *exec.Cmd; LastRequest atomic.Int64; cancel context.CancelFunc; exitC chan struct{}; proxy *httputil.ReverseProxy }`. Define `type Supervisor struct { mu sync.Mutex; children map[string]*Child; spawnGroups map[string]*sync.Once; binPath string; idleTimeout time.Duration; spawnTimeout time.Duration }` with per-project `sync.Once` to handle concurrent cold-spawn requests without herd contention on the main mutex. Implement `NewSupervisor(opts Options) *Supervisor` with `binPath = os.Executable()` by default. Implement `(*Supervisor) GetOrSpawn(ctx context.Context, projectID, projectDir string) (*Child, error)` which spawns via `exec.CommandContext(cancelCtx, s.binPath, "--project-dir", projectDir, "_serve-child", "--port", "0")`. **The arg order is load-bearing**: `--project-dir` is a persistent flag on the root cobra command (main.go:37) and must appear BEFORE the `serve` subcommand, not after, or cobra rejects it (feasibility critic A8). Capture stdout via `cmd.StdoutPipe()`, call `cmd.Start()`, launch a goroutine that scans stdout with `bufio.Scanner` for exactly one line matching `htmlgraph-serve-ready port=<N> pid=<PID>` with a 5s deadline. After the handshake, the scanner goroutine drains the pipe into `io.Discard` forever so the child doesn't block — though the child should have already redirected its own stdout to a log file per slice 1's requirement. Populate `Child.Port` and `Child.proxy` (pre-built `httputil.NewSingleHostReverseProxy` pointed at `http://127.0.0.1:<port>` with `FlushInterval = 100 * time.Millisecond` for SSE), and return. Second goroutine per child calls `cmd.Wait()`, closes `exitC`, removes from the map under the mutex. Implement `Shutdown(ctx)` which SIGTERMs all children in parallel, waits with deadline, SIGKILLs survivors. Implement `RunIdleReaper(ctx)` ticking every 60s. The closest prior-art pattern is `internal/planchat/backend.go:174` (exec.CommandContext + StdoutPipe + Start + scan + Wait) — model the scanner goroutine after it. Use `os.Executable()` not `os.Args[0]` (plan_cmds.go:268 uses os.Args[0] which is wrong for moved binaries / symlinks).
Why
The supervisor is the isolation boundary. All OS-level process concerns live in one place: spawning, port handshake parsing, crash detection, idle reaping, zombie prevention on shutdown. Isolating this behind a clean interface lets slice 3's reverse-proxy layer ignore process-level concerns and lets tests verify lifecycle in isolation. Using `os.Executable()` guarantees version parity between parent and child (no PATH resolution, no split-binary installs) and plays well with `go test` (tests build their own binary and spawn it).

#3 Replace runServer body with parent+proxy; drop --global flag; current-project auto-detect M Med Pending
What
This is the slice where `htmlgraph serve` transitions from 'single-project server' to 'parent with reverse proxy and projects-landing'. Multi-part change: **(1) Drop the `--global` flag entirely.** Delete the flag definition from the serve cobra command. `htmlgraph serve --global` becomes an unknown-flag error (cobra default). Users who typed `--global` get a clear cobra error and can re-run without the flag. **(2) Merge `runGlobalServer` body into `runServer`.** `runServer` now always starts the parent process: builds the global mux (landing SPA + `/api/mode` + `/api/projects` + `/p/<id>/*` proxy), instantiates the `childproc.Supervisor` from slice 2, starts the idle reaper goroutine, and calls `http.ListenAndServe(addr, mux)` on the configured `--port` (default 8087). The old single-project behavior is gone from `runServer` — it moved to the hidden `_serve-child` command in slice 1. Delete the now-dead `runGlobalServer` function. **(3) Proxy handler registration.** Register `mux.Handle("/p/", corsMiddleware(http.HandlerFunc(proxyHandler)))` where `proxyHandler` parses `/p/<project-id>/<rest>`, validates the project ID (reject empty, `.`, `..`, `/`, null bytes, non-alphanumeric+underscore+hyphen — returns 400 on validation failure), looks up the project in the registry to get `ProjectDir` (returns 404 if unknown), calls `supervisor.GetOrSpawn(r.Context(), projectID, projectDir)` (returns 502 on spawn failure), strips the `/p/<id>` prefix from `r.URL.Path` via `path.Clean("/" + rest)` not string manipulation, calls `child.LastRequest.Store(time.Now().Unix())`, and forwards via the pre-built `child.proxy` (already has `FlushInterval = 100 * time.Millisecond` for SSE from slice 2). The proxy's `ErrorHandler` (set at construction time in slice 2) returns 502 on child unreachable. **(4) Current-project auto-detection (Option B).** Before starting the HTTP server, call `findHtmlgraphDir()` to check if CWD contains a `.htmlgraph/` directory. If yes, derive the project ID via `SHA256(absPath)[:8]`, auto-register it in the registry if not already present (same `registry.Upsert` call the `PersistentPreRunE` uses), and extend the startup banner to print BOTH URLs: `http://localhost:<port>/` (doorway) and `http://localhost:<port>/p/<id>/` (current project deep link). If CWD has no `.htmlgraph/` or the registration fails, silently print only the doorway URL. The detection code is ~10 lines at the top of `runServer` before the mux is built. Do NOT auto-open the browser — just print the URL for the user to click. **(5) Graceful shutdown.** Plumb `supervisor.Shutdown(ctx)` through `http.Server.RegisterOnShutdown` so when the parent receives SIGINT/SIGTERM, it cleanly kills all children before exiting.
Why
The reverse proxy is the one-line contract between the global server's outer mux and the per-project children. It's the only layer that needs to understand 'how to get a request from the user into a child'; everything else (child lifecycle, frontend routing, aggregate reads) is independent. The `FlushInterval` setting is load-bearing for the SSE event stream (without it, `httputil.ReverseProxy` buffers the response and breaks EventSource). The 502 ErrorHandler is the crash-recovery story: if the child dies mid-request, the user sees an error, refreshes, and the next request spawns a fresh child.

Gaps & Missing Considerations
  • Gap Child stdout/stderr logging after handshake — where do child logs go? If parent's pipe reader stops reading after handshake, child will eventually block on pipe write. Need per-project log file or /dev/null redirect.
  • Gap Dashboard UX during cold-spawn latency (~300-500ms). First click on an idle-reaped project shows blank page briefly. Consider a loading indicator or pre-warm on landing render.
  • Gap Herd problem — if N concurrent requests hit an un-spawned project, all N will wait on the supervisor mutex. Per-project condition variable or singleflight pattern keeps spawn serialized without blocking the N requests unnecessarily.
  • Gap Auth/ACL — no enforcement today. Under new model a user navigating /p/<unknown-id>/ gets 404, /p/<known-id>/ gets full access. Not a regression but should be explicitly acknowledged in design constraints.
  • Gap Version skew — two global servers (e.g., old dev binary + new marketplace binary) on the same machine will fight over ephemeral ports and can leave orphaned children. Must document 'only one global server per machine' or add a lockfile.
  • Gap Plan template path routing — plan HTML files are served at /plans/<id>.html via FileServer (serve.go:80), not via the API router. Under new architecture this path is only accessible via /p/<id>/plans/<id>.html through the child proxy. Slice 5 must account for this.
Alternative Approaches
  • Alt Worker pool (N fixed processes handling any project on-demand) vs per-project child. Plan correctly chose per-project because hook/env context must be baked at process start. Correct tradeoff.
  • Alt In-process goroutine pools with DB sharding. Rejected because a panic in project A could crash project B. Plan correctly prioritized crash isolation.
  • Alt Unix domain sockets instead of TCP. Eliminates port discovery but adds Unix-only complexity. TCP loopback is the right choice for cross-platform scope.

Synthesis

Plan was critiqued at v2 and revised at v3 (true doorway decision). At v4, the human dropped the `--global` mode flag entirely: `htmlgraph serve` becomes the only human-facing serve command and always starts the parent with reverse proxy. Single-project serve lives on as a hidden `_serve-child` cobra command invoked exclusively by the supervisor to spawn children. When run from inside a project directory, the parent auto-detects the current project and prints both the doorway URL and a deep link to `/p/<id>/` in the startup banner — Option B from the UX discussion. This simplifies the user-facing command surface (one command, no mode flag) while preserving the architectural boundary (parent holds zero DB handles, child lifecycle in one place, reverse proxy as the only entry into per-project state). The two falsified assumptions from v2 (A5 absolute-path URLs, A8 persistent-flag arg order) remain fixed in slices 5 and 2. The three mechanical corrections (A2 autoIngestOnce, A3 combo-handler surgery, A7 dashboardFS) remain applied in slice 1 and slice 4. Slice 4 is now purely architectural cleanup — no aggregate reads, no `projectCache`, no WAL pragma audit, grep-enforceable zero-DB-handles invariant. Slice 5 rewires the plan template fetches via a template-injected `window.htmlgraphPlanBase` variable. Slice 6 proves the boundary with an integration test that writes to projectA and reads projectB's SQLite directly. Verdict: BUILDABLE AS WRITTEN at v4. All 5 questions answered.
