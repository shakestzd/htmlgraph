HtmlGraph Design Slices Decisions Critique Progress Chat

Plan: HTML-canonical architecture migration

Lock in a public HTML schema (JSON-LD payload + microdata body + link rel head edges + meta-name primitives + time datetime) that makes the work-item HTML files the canonical queryable substrate. Migrate every consumer (CLI, dashboard, ingest pipeline) off SQLite as a system requirement, in six phases. SQLite is allowed back later strictly as a disposable derived speed tier — not in this plan.

DesignPending

Problem

`.htmlgraph/htmlgraph.db` has corrupted three times in nine days. The architecture claims SQLite is a "derived read index" but the team treats it as authoritative — corruption is currently a session-breaking incident rather than a transparent rebuild. Today the OTEL activity feed shows zero spans because the DB is empty (no schema), even though 198 spans are durably written to NDJSON for the current session. Meanwhile the CLI is already ~85% HTML-direct (per a read-path audit): only `trace`, `lineage`, `session list`, and a supplemental block in `status` actually require SQLite. The cost-to-benefit has flipped: SQLite imposes a corruption class on every session for marginal speed wins on three commands at current scale.

Goals

  1. **Eliminate the corruption class** — losing the SQLite file must become a sub-second nuisance (rm + rebuild from canonical sources), not an incident.
  2. **Promote HTML to genuinely canonical** — every fact in a work-item file is queryable by an explicit, versioned schema (JSON-LD, microdata, `<link rel>`, `<meta name>`, `<time datetime>`).
  3. **Remove SQLite as a system requirement** — the binary builds, ships, and runs without any SQLite driver; full test suite passes with `--block-sqlite`.
  4. **Preserve user-facing CLI surface** — every existing command keeps its name, flags, and behavior; users don't notice the migration.
  5. **Land a new `htmlgraph query` subcommand** — jq-style filter over JSON-LD payloads, replacing ad-hoc shell pipelines.

Constraints

  • No new runtime language deps (no Python, no Rust binary). Pure-Go embeds only (e.g. `itchyny/gojq`).
  • Schema vocabulary is versioned at `https://htmlgraph.dev/v1/`; future breaking changes go to `/v2/` plus a migrator.
  • Phase 0 must land first — we do not remove the SQLite path while it's the only thing serving the activity feed.
  • Every SQLite removal step must keep a non-SQLite fallback exercised by CI; no quiet code paths.
  • Migration of existing `.htmlgraph/*.html` files must preserve content semantics — diff-tested before rollout.

#5 Phase 2a — Templater refactor: derive all schema layers from one source struct L High Pending
What
Audit every code path that emits work-item HTML (work item create/edit, plan rewrite, track render, `commitPlanChange`, session HTML emission) and refactor each to render from a single `internal/schema/v1.<Type>` struct. The renderer derives, from that one struct: (a) the JSON-LD `<script>` payload, (b) `<meta name="htmlgraph:*">` head metas, (c) `<link rel="...">` head edges, (d) microdata `itemscope`/ `itemprop` decoration of body elements, (e) `<time datetime>` for every timestamp. No fact may be hand-authored in two layers — the struct is source of truth, the template derives. Every write path runs `schema.Validate` on its output and refuses to write malformed HTML.
Why
Removes the duplication-and-drift class permanently. Once this lands, "the HTML schema" is a real architectural invariant rather than a style guide. Validation at write time turns schema regressions into test failures rather than corrupted artifacts.
Files
internal/workitem/render.go, internal/plan/render.go, internal/track/render.go, internal/session/render.go, internal/htmltemplate/derive.go, cmd/htmlgraph/feature_create.go, cmd/htmlgraph/feature_edit.go, cmd/htmlgraph/plan_rewrite.go
Done when
  • Every emitter calls a single `htmltemplate.Render(struct)` entry point.
  • Every emitter validates output via `schema.Validate` before writing; CI test asserts no emitter bypasses the validator.
  • Generated HTML for a sample Feature struct contains JSON-LD with the same `status` as the rendered `<meta name>` and the body microdata; mismatch is impossible by construction.
Test Strategy
Unit: TestRender_FeatureMatchesSchema — render a sample Feature, validate output, assert specific selectors. Unit: TestRender_NoDuplication — given a struct with `status="x"`, mutate the struct and re-render; assert all five layers updated. Integration: TestEmitterWriteRefusesInvalid — corrupt a render path to emit a missing-field HTML; assert write returns error. Regression: existing `internal/workitem/*_test.go`, plan render tests, and snapshot tests still pass.

#6 Phase 2b — Backfill migration of existing `.htmlgraph/*.html` files L High Pending
What
Implement `htmlgraph migrate-html --to v1`: iterate every file under `.htmlgraph/{features,bugs,spikes,plans,tracks,sessions}/*.html`, extract facts via the current selector-based parser, populate a `schema.v1.<Type>` struct, re-render through the new templater (slice-5), and write back. Diff-test mode (`--dry-run`) prints semantic diffs (status, edges, timestamps) without writing. Failures are logged; the command exits non-zero if any file cannot be cleanly migrated. Add a one-off CI job that runs the migrator against a fixture corpus and asserts it converges.
Why
Without backfill, the codebase has two HTML formats: schema-compliant (new emissions) and pre-schema (existing). The dual-format burden defeats the whole plan. Backfill collapses the world to a single format before Phase 3 reads can rely on it.
Files
cmd/htmlgraph/migrate_html.go, internal/migrate/v0_to_v1.go, internal/migrate/v0_to_v1_test.go, internal/migrate/testdata/legacy/
Done when
  • `htmlgraph migrate-html --dry-run` produces a clean diff for the working repo's `.htmlgraph/` (no semantic content lost).
  • After running `htmlgraph migrate-html --to v1`, `internal/schema.Validate` returns nil for every file under `.htmlgraph/`.
  • CI runs the migrator against `internal/migrate/testdata/legacy/` and asserts byte-equality on a second run (idempotent).
Test Strategy

Integration: TestMigrate_FullCorpus — migrate the real `.htmlgraph/` (in CI sandbox), assert all files validate.

#7 Phase 3a — `internal/htmlparse` JSON-LD fast path + ParseLinks + ParseHeadMeta M Low Pending
What
Add three new exported functions to `internal/htmlparse`: (1) `ParseJSONLD(r) (*schema.v1.<Type>, error)` — extracts the first `<script type="application/ld+json">` and unmarshals. (2) `ParseLinks(r) ([]Link, error)` — head-only stream parse for `<link rel="..." href="...">` tags. (3) `ParseHeadMeta(r) (map[string]string, error)` — head-only stream parse for `<meta name="htmlgraph:*" content="...">`. All three short-circuit on `</head>`. Existing `ParseFile` continues to work via goquery for legacy callers and body-content extraction. Benchmark JSON-LD parse vs full parse and document the win.
Why
The fast paths are the perf justification for not needing SQLite for the common queries. ~1ms-per-file JSON-LD parse vs ~10ms goquery means a 200-item snapshot is ~200ms instead of 2 seconds — acceptable for CLI use, generous for dashboard cold-start.
Files
internal/htmlparse/jsonld.go, internal/htmlparse/links.go, internal/htmlparse/head_meta.go, internal/htmlparse/jsonld_test.go, internal/htmlparse/links_test.go, internal/htmlparse/head_meta_test.go, internal/htmlparse/bench_test.go
Done when
  • All three functions return correct results on every fixture from slice-4.
  • Head-only parsers stop at `</head>` (verified by benchmarking a 1MB body — parse time independent of body size).
  • Documented benchmark in `bench_test.go`: JSON-LD path ≥5× faster than goquery on a real Feature file.
Test Strategy
Unit: TestParseJSONLD_AllFixtures — table-driven over schema fixtures. Unit: TestParseLinks_HeadOnly — file with malicious gigantic body parses in < 5ms. Bench: BenchmarkParseJSONLD vs BenchmarkParseFile_GoQuery on a real Feature.

#8 Phase 3b — Migrate `trace`, `lineage`, `session list`, and `status` off SQLite L High Pending
What
Rewrite four CLI command read paths to use HTML + git directly: (1) `htmlgraph trace <id>`: scan commits via `git log --grep`/path- filter; read produced-by-session and produced-commit `<link rel>` edges from work-item heads. (2) `htmlgraph lineage <id>`: BFS over work-item heads using `htmlparse.ParseLinks`; resolve neighbor IDs via `<link rel>` href basenames. (3) `htmlgraph session list`: scan `.htmlgraph/sessions/*.html` head metas; sort by `htmlgraph:created`. (4) `htmlgraph status`: drop the `dbpkg.CommitAttributionRate` block or recompute it on demand from work-item JSON-LD. All four commands gain a `--block-sqlite` debug flag (process-level) that fails if any SQLite open is attempted. CI runs every command with the flag set and asserts success.
Why
These four are the only SQLite-dependent CLI paths per the read-path audit. Moving them off SQLite is the bulk of the work that lets Phase 4 actually delete the dependency. The `--block-sqlite` flag is the binding contract — without it, "we removed SQLite" is aspirational.
Files
cmd/htmlgraph/trace.go, cmd/htmlgraph/lineage.go, cmd/htmlgraph/session.go, cmd/htmlgraph/status.go, cmd/htmlgraph/block_sqlite.go, cmd/htmlgraph/trace_test.go, cmd/htmlgraph/lineage_test.go, cmd/htmlgraph/session_test.go
Done when
  • `trace`, `lineage`, `session list`, and `status` all run successfully with `--block-sqlite`.
  • Output of each command is byte-identical (or semantic-equivalent, documented) to the SQLite-backed version on a representative fixture project.
  • CI runs the full CLI test suite with `HTMLGRAPH_BLOCK_SQLITE=1` and asserts no command opens a DB.
Test Strategy
Unit: TestTrace_NoSQLite — sample fixture project, compare output to recorded golden. Unit: TestLineage_NoSQLite — fixture with 3-deep edges, BFS produces correct order. Integration: TestCLI_BlockSQLite — run `find`, `relevant`, `snapshot`, `wip`, `feature show`, `trace`, `lineage`, `session list`, `status`, `analytics` with the flag; all exit 0. Regression: existing trace/lineage/session-list tests still pass.

#9 Phase 3c — `htmlgraph query` subcommand (gojq over JSON-LD) M Low Pending
What
Add `htmlgraph query <jq-expr> [paths...]`: extracts JSON-LD from every HTML file in the given paths (default `.htmlgraph/**/*.html`), pipes them through an embedded `github.com/itchyny/gojq` filter, and emits JSONL on stdout. Supports `--type feature` shorthand to scope to a single work-item type. Default output is one JSON object per file; `--slurp` collects into an array. Document common recipes in `docs/cli/query.md` (find all P1 bugs, lineage tree as JSON, etc.).
Why
Gives users and downstream scripts a stable shell-scriptable interface that doesn't require a new CLI subcommand for every read pattern. Replaces ad-hoc shell pipelines and prevents feature-creep pressure on the core CLI surface. Also serves as a power-user escape hatch when the dashboard doesn't expose a needed query.
Files
cmd/htmlgraph/query.go, cmd/htmlgraph/query_test.go, docs/cli/query.md, go.mod, go.sum
Done when
  • `htmlgraph query '.status' --type feature` lists the status of every feature file.
  • `htmlgraph query '. | select(.status == "in_progress")'` filters correctly.
  • Embedded gojq does not reach the network; runs offline.
  • Documented at least 5 recipes in `docs/cli/query.md`.
Test Strategy
Unit: TestQuery_TypeFilter — table-driven against schema fixtures. Unit: TestQuery_SelectExpression — assert correct subset. Integration: TestQuery_RealRepo — runs against `.htmlgraph/` in CI; expected output count matches `find | wc -l`.

#10 Phase 3d — Dashboard endpoints serve JSON-LD; activity feed reads NDJSON directly L High Pending
What
Replace the SQLite-backed dashboard read path: (1) New endpoint `GET /api/items/:id` streams the JSON-LD payload from the corresponding HTML file. (2) New endpoint `GET /api/manifest` returns `[{file, mtime, type, id}, ...]` — head-only scan, cached with mtime invalidation. (3) `GET /api/feed` is rewritten to stream `.htmlgraph/sessions/*/events.ndjson` directly; merges hook events and OTEL spans into one timestamp-sorted stream. The activity feed bug (this conversation's symptom) goes away by construction — both kinds of events come from the same source. (4) Old SQLite-backed feed endpoint kept temporarily behind `?legacy=1` for parity comparison; removed in slice-11.
Why

#11 Phase 4 — Delete SQLite from the codebase M Med Pending
What
With slices 1-10 landed, remove SQLite as a system requirement: (1) Delete `internal/storage/`, `internal/db/`, `cmd/htmlgraph/cache.go` (cache eviction is SQLite-specific), `cmd/htmlgraph/dbpath.go`. (2) Remove the SQLite driver (`mattn/go-sqlite3` or whichever is in `go.mod`); run `go mod tidy`. (3) Remove `htmlgraph reindex` (no index) — keep `htmlgraph recover-db` only if it now means "rebuild any cached query state"; otherwise delete it too. (4) Update `AGENTS.md`, `CLAUDE.md`, plugin system prompts: remove "SQLite read index" from the architecture table; replace with "HTML files (canonical) + NDJSON sessions (canonical)". (5) Update `.htmlgraph/.gitignore` to permanently ignore `htmlgraph.db*`; add a deploy-script step that nukes the file if present. (6) Verify binary size shrinks; record before/after in commit message.
Why
Closes the migration. Until SQLite is actually deleted, the team will keep accidentally adding code that reaches for it. "Removing the option" is the only way to make the architecture promise real.
Files
internal/storage/, internal/db/, cmd/htmlgraph/cache.go, cmd/htmlgraph/cache_pollution_test.go, cmd/htmlgraph/dbpath.go, go.mod, go.sum, AGENTS.md, CLAUDE.md, cmd/htmlgraph/prompts/system-prompt.md, .htmlgraph/.gitignore, scripts/deploy-all.sh
Done when
  • `go.sum` no longer references `mattn/go-sqlite3` or any SQLite driver.
  • `grep -r 'database/sql' cmd/ internal/` returns zero matches.
  • Full test suite passes.
  • Binary size reduction documented in commit message (expected: ~3-5 MB).
  • Architecture table in `AGENTS.md` no longer mentions SQLite.
Test Strategy
Integration: TestNoSQLiteDriver — `go test ./...` with a build tag that fails on `database/sql` import. Regression: full CLI command matrix passes. Smoke: fresh checkout, `go build`, `htmlgraph status` against a clean repo — works.
