HtmlGraph Design Slices Decisions Critique Progress Chat

Plan: Per-session OTel collector with NDJSON ingest + SQLite as derived index

Move the OTLP receiver out of htmlgraph serve into per-session collector subprocesses that append OTel signals to session-scoped NDJSON files, and build an NDJSON-to-SQLite indexer in serve to preserve existing dashboard APIs. Retires the shared-port shared-writer model that causes parallel-session merging (bug-e8b8900b), SQLITE_BUSY cascades on the OTel write path (bug-08681166 class), and serve-restart span drops. Establishes the NDJSON-canonical + SQLite-derived pattern without committing to full SQLite removal (deferred to follow-on plans).

Problem

The OTLP receiver is embedded inside htmlgraph serve and writes to the shared .htmlgraph/htmlgraph.db SQLite file on a project-hashed port (4318 + FNV32a(projectDir) % 1000, internal/otel/receiver/port.go:11). This creates four recurring failure modes. (1) Parallel htmlgraph claude terminals in the same project share one port and one writer; Claude Code's span tagging is the only thing distinguishing their sessions, and bug-e8b8900b documents session merging from this. (2) The receiver competes with hook processes for SQLite writes, producing SQLITE_BUSY cascades — bug-08681166 (in-progress), and the already-resolved bug-6ab963d3 and bug-59b98285, all describe the same contention class. (3) Restarting serve drops in-flight spans because Claude Code's OTel SDK has no durable retry against a closed port (claude_serve_autostart.go:93 logs a warning and accepts the loss). (4) Session identity is authoritative only in Claude Code — HTMLGRAPH_SESSION_ID leaks from parent shells into subprocesses (internal/hooks/runner.go:187), misattributing hook events in subagent contexts.

Goals

  1. **Per-session isolation** — each htmlgraph claude/yolo/dev session owns its own collector subprocess, ephemeral port, and NDJSON file; parallel terminals never contend at the ingest layer.
  2. **Eliminate OTel writer collisions with SQLite** — collectors never hold a SQLite write lock directly; the bug-08681166 class of failures stops reaching the OTel ingest path entirely.
  3. **Lifecycle independence from serve** — collectors survive serve restarts without dropping spans; serve becomes a pure reader + semantic-op server.
  4. **Launcher-owned session identity** — htmlgraph mints the session ULID and binds it to every emitted event, replacing Claude Code's span-tagged session.id as the authority.
  5. **Zero dashboard regression** — every /api/otel/* endpoint keeps returning the same data shape; the NDJSON→SQLite indexer preserves all existing query paths.

Constraints

  • Keep SQLite as the dashboard read index for this plan. Removing it entirely is a future phase that depends on answers we do not yet have for six genuine-SQL sites: yolo-guard LIKE scans (yolo_guard.go:539–704), DSL graph traversal (internal/graph/dsl.go), feature_files self-join overlap (feature_files_repo.go:86), claims CAS and lease expiry (claim_repo.go), OTel span re-parenting transactions (writer.go:553), and dashboard sessions list 5-table correlated subqueries (api.go:116–146).
  • Reuse the existing syscall.Flock pattern (session_html.go:147, materialize.go:241) for NDJSON atomicity on oversize writes. Do not introduce a new lock primitive.
  • PIPE_BUF atomicity is 4096 bytes on Linux and 512 bytes on macOS. Some transcript_assistant.go:267 payloads exceed 4KB. Plan must name an explicit size-overflow strategy, not assume single-syscall atomicity.
  • Serve auto-spawn must continue to work for semantic ops and the dashboard — only the OTLP receive-side is removed from serve. The existing checkServeLock/writeServeLock/spawnDetachedServe pattern (claude_serve_autostart.go:179–255) is reused for the dashboard path.
  • Dashboard is an SPA (plugin/static/dashboard/js/app.js, 3435 lines) with substantial client-side aggregation already. The indexer preserves current API response shapes; no frontend rewrite is in scope.
  • Launcher currently detaches subprocesses via Setpgid:true and uses PID lockfiles. Collector subprocess reuses these mechanisms. Linux-only PDEATHSIG is not required if SIGTERM-on-exit is wired reliably at layered levels (SessionEnd hook + deferred launcher SIGTERM + idle timeout).

#1 Extract SignalSink interface; add NDJSONSink implementation M Low Pending
What
Refactor internal/otel/receiver/writer.go to depend on a SignalSink interface with WriteBatch(ctx, *otel.UnifiedSignalBatch) error. Keep the current SQLite-backed writer as the first concrete implementation under internal/otel/sink/sqlite/. Add internal/otel/sink/ndjson/ that opens .htmlgraph/sessions/<sid>/events.ndjson (layout per Q1) in O_APPEND mode, serializes each UnifiedSignal as one JSON line, and handles payloads larger than PIPE_BUF according to the strategy chosen in Q4. Both sinks implement the same interface and are swappable at construction time. internal/otel/receiver/receiver.go's New() accepts a SignalSink rather than opening a *sql.DB directly.
Why
Today writer.go is deeply SQLite-coupled: five prepared statements, BEGIN IMMEDIATE transactions, cross-table FK resolution against pending_subagent_starts and active_work_items. Without an interface boundary, a per-session collector cannot choose between sinks and cannot be tested without a real SQLite file. This interface extraction is the prerequisite that unblocks every downstream slice. Supports goal "Eliminate OTel writer collisions" by making sink selection explicit.

#2 htmlgraph otel-collect subcommand S Low Pending
What
New CLI subcommand `htmlgraph otel-collect --session-id <sid> --project-dir <path> --listen :0`. On start, constructs a receiver.Receiver wired to NDJSONSink rooted at .htmlgraph/sessions/<sid>/, chooses an ephemeral port via net.Listen(":0"), prints the chosen port to stdout in a single line `listen=127.0.0.1:<port>`, then blocks on Receiver.Start(ctx). Handles SIGTERM gracefully: flush pending batch (bounded ≤3s), close sink, exit 0. Emits a collector_start NDJSON line on boot: {"type":"collector_start","htmlgraph_sid":"<sid>","pid":<pid>, "port":<port>,"ts":<ns>} so the indexer can correlate collector lifetime with session events.
Why
The collector must run as a separate process to achieve lifecycle independence from serve. Printing the chosen port on stdout is the simplest contract between launcher and collector — no advertisement protocol, no lockfile scan. SIGTERM drain is the only shutdown path that preserves in-flight spans. Forcing the launcher to pass session-id via flag closes the HTMLGRAPH_SESSION_ID env-leak misattribution at runner.go:187. Supports goal "Launcher-owned session identity".
Files
cmd/htmlgraph/otel_collect.go, cmd/htmlgraph/main.go
Done when
  • `htmlgraph otel-collect --session-id 01HK... --project-dir /tmp/proj --listen :0` prints a single `listen=127.0.0.1:<port>` line on stdout within 200ms of start.
  • POSTing an OTLP protobuf to the printed port produces a corresponding NDJSON line in .htmlgraph/sessions/01HK.../events.ndjson within 500ms.
  • SIGTERM drains the pending batch and exits 0 within 3s; SIGKILL accepted without corrupting NDJSON beyond the last partial line (tolerable by indexer's checkpoint logic).
  • collector_start event is the first line of the NDJSON file and contains htmlgraph_sid, pid, port, ts.
Test Strategy
Unit: cmd/htmlgraph/otel_collect_test.go — spawn subprocess via exec.Command, read port line, POST OTLP protobuf, kill SIGTERM, assert NDJSON contents. Integration: parallel launch — spawn 3 otel-collect processes with different session IDs in the same project, assert each gets a distinct port and writes to its own NDJSON file with no cross-contamination. Regression: `htmlgraph --help` output includes otel-collect; no other subcommand is affected.

#3 Launcher spawns per-session collector child M Med Pending
What
In cmd/htmlgraph/claude.go:launchClaude, before buildClaudeLaunchEnv: (1) generate a ULID for HTMLGRAPH_SESSION_ID; (2) exec `htmlgraph otel-collect --session-id <sid> --project-dir <root> --listen :0` as a child with Setpgid:true, matching claude_serve_autostart.go:255; (3) scan stdout for the `listen=` line with a 3s timeout; abort launch with a clear error if the collector fails to bind; (4) set OTEL_EXPORTER_OTLP_ENDPOINT=http://<address> and HTMLGRAPH_SESSION_ID=<sid> in the launch env, replacing the current static port derivation at claude_env.go:77; (5) register a deferred SIGTERM to the collector PID on launchClaude return, with the collector PID written to .htmlgraph/sessions/<sid>/.collector-pid so SessionEnd hook can also target it (slice 7). The existing MaybeShowOtelNotice and ensureServeForOtel calls stay intact for this slice — slice 6 renames and re-scopes them.
Why
The launcher is the only process that lives exactly as long as one Claude Code session. Spawning the collector here is what makes "session ID owned by the launcher" a code-level fact, not just documentation. Capturing the ephemeral port from child stdout is the irreversible step — once claude-code is exec'd with a given OTel endpoint, it cannot be reconfigured at runtime. Fail-fast on bind failure prevents orphaned claude-code processes exporting telemetry to nowhere. Directly addresses the parallel-terminal merging root cause (bug-e8b8900b).

Done when

  • Running htmlgraph claude in three terminals in the same project produces three distinct otel-collect children on three distinct ports writing to three distinct NDJSON files with no cross-session events mixed.
  • Collector bind failure (e.g. no ports available) aborts launchClaude with a non-zero exit and a clear error message; no orphaned claude-code process is exec'd.

#5 NDJSON→SQLite indexer loop inside serve L Med Pending
What
New internal/otel/indexer/ package that watches .htmlgraph/sessions/*/events.ndjson files (fsnotify on Linux, polling every 500ms on macOS per Q2), tails each file, parses one JSON line at a time, and applies it to SQLite through the SQLite SignalSink from slice 1. Maintains a per-file checkpoint in .htmlgraph/sessions/<sid>/.index-offset (written atomically via write-to-temp-then-rename after each successful batch commit). On indexer boot, replays from last checkpoint to end-of-file for every session directory. Mount the indexer in cmd/htmlgraph/serve_child.go alongside the existing autoIngestLoop, feature-flagged via HTMLGRAPH_INDEXER=1 if Q5 selects parallel-validation rollout, or unconditionally on if Q5 selects cutover.
Why
With collectors writing NDJSON instead of SQLite directly, every /api/otel/* endpoint (backed by otel_signals, otel_resource_attrs, otel_session_rollup) would break on day one. The indexer is the bridge: NDJSON is canonical, SQLite becomes derived and rebuildable. Checkpoint files are what make "rebuild by deleting htmlgraph.db and restarting serve" a single-command recovery — serve replays every NDJSON from offset 0 and repopulates the index. Directly supports goal "Zero dashboard regression".
Files
internal/otel/indexer/indexer.go, internal/otel/indexer/checkpoint.go, internal/otel/indexer/watch_linux.go, internal/otel/indexer/watch_fallback.go, cmd/htmlgraph/serve_child.go
Done when
  • Indexer starts inside htmlgraph serve, tails every .htmlgraph/sessions/*/events.ndjson, applies new NDJSON lines to SQLite, and advances .index-offset atomically after each committed batch.
  • Killing serve mid-replay and restarting resumes from the last checkpointed offset with zero duplicate rows (INSERT OR IGNORE semantics preserved by the SQLite sink).
  • Full-rebuild test: delete htmlgraph.db, restart serve, assert otel_signals row count equals total parseable NDJSON line count across all sessions.

#6 Strip OTLP receiver from serve; rename ensureServeForOtel M Med Pending
What
Remove receiver.New/Start calls from cmd/htmlgraph/serve_child.go. The embedded OTLP endpoint on the project-hashed port is deleted. Rename claude_serve_autostart.go:ensureServeForOtel to ensureServeForDashboard and change its semantics — serve is no longer auto-spawned for telemetry reasons. It is only auto-spawned when (a) a CLI command requires semantic-op execution or (b) the user explicitly opens the dashboard. Why
This is the slice that actually shuts the contention class down: once the OTLP receiver is gone from serve and collectors are the only ingest path, bug-08681166 (busy_timeout lost) stops reaching OTel writes entirely. The rename clarifies serve's role as a reader plus semantic-op singleton, not a telemetry endpoint. Deleting port.go eliminates the multi-project-collision history documented in bug-d2ba2a82. Directly supports goals "Eliminate OTel writer collisions" and "Lifecycle independence from serve".

#7 Session cleanup: graceful drain + retention hygiene M Low Pending
What
Two pieces. (1) On SessionEnd hook, read .htmlgraph/sessions/<sid>/.collector-pid (written by slice 3), send SIGTERM, wait up to 3s for clean drain, fall back to SIGKILL if needed. This is Q3's "primary" layer. (2) Retention job in a new internal/otel/retention/ package: on serve startup and every 24h thereafter, walk session directories, detect sessions whose sessions.status = 'done' is older than HTMLGRAPH_SESSION_RETAIN_DAYS (default 30), archive events.ndjson into .htmlgraph/archive/<yyyy-mm>/<sid>.tar.zst, delete the live session dir (but preserve the indexer-populated SQLite rows — those age out via existing cleanup commands). Dry-run mode via HTMLGRAPH_RETENTION_DRYRUN=1 logs intended actions without moving files.
Why
Without SessionEnd SIGTERM, collectors either persist past session lifetime (waiting for a launcher SIGTERM that may never fire after launcher SIGKILL), or get reaped by OS cleanup with spans still in their batch buffer — losing them. Without retention, .htmlgraph/sessions/ grows unbounded over months. Restoring an archived session is supported so lineage queries over historical transcripts still work. This slice closes the correctness gap named in Q3 and the operational gap noted in the design doc.
Files
internal/hooks/session_end.go, internal/otel/retention/retention.go, cmd/htmlgraph/serve_child.go, cmd/htmlgraph/session_restore.go
Done when
  • SessionEnd hook reads .collector-pid, sends SIGTERM, waits up to 3s, falls back to SIGKILL; the collector flush completes and the final NDJSON batch has no torn last line.
  • Retention job runs at serve boot and on a 24h timer; HTMLGRAPH_RETENTION_DRYRUN=1 logs intended archive/delete actions without moving files.
  • A session archived to .htmlgraph/archive/ is restorable via `htmlgraph session restore <sid>`; the indexer picks up the restored NDJSON on next replay without duplicating rows.
Test Strategy
Unit: internal/otel/retention/retention_test.go — construct 10 synthetic session directories with varying ages, run retention, assert correct archive and deletion behavior, assert live sessions (status != 'done') are untouched. Integration: SessionEnd signal — spawn collector, write 100 signals, invoke SessionEnd hook, assert collector exits 0 within 3s and the final batch is fully flushed to NDJSON. Integration: restore flow — archive a session, run `htmlgraph session restore <sid>`, assert indexer picks up the restored file and otel_signals row count matches pre-archive value. Regression: existing session-end flows (transcript sync, feature work-on capture) still fire in the correct order relative to the new SIGTERM step.
