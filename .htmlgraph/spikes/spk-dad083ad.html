## Problem Dashboard activity feed flickers/resets expanded state when WebSocket triggers a live update. Need to understand the full update mechanism before implementing a fix. ## Architecture: Two Dashboards Exist There are TWO dashboard implementations in this codebase: 1. src/python/htmlgraph/dashboard.html - Uses WebSocket (/ws/events) for live updates - Activity feed is the "conversation-feed" style (conversation turns with toggle triangles) - insertNewEventIntoActivityFeed() does surgical DOM insertion — NO full reload - No table structure; uses flex divs 2. src/python/htmlgraph/api/templates/dashboard-redesign.html - Uses SSE (/activity-feed/stream) for live updates - Activity feed is a proper HTML table (activity-feed-hierarchical.html) - refreshActivityFeed() does FULL RELOAD via htmx.ajax(GET, /views/activity-feed) - Has save/restore logic for expanded state (window._htmxFeedExpandedState) The FLICKER issue is in dashboard-redesign.html (the active one served by FastAPI). ## Live Update Mechanism (dashboard-redesign.html) SSE-based flow: 1. initActivityFeedSSE() connects to /activity-feed/stream (SSE endpoint) 2. Server sends "feed-update" events when new DB events arrive 3. Client calls debouncedFeedRefresh() (800ms debounce) 4. debouncedFeedRefresh() calls refreshActivityFeed() 5. refreshActivityFeed() calls htmx.ajax(GET, /views/activity-feed, {target: #content-area}) 6. This does a FULL innerHTML replace of #content-area with fresh server-rendered HTML 7. The old delta approach (/views/activity-feed/delta?since=) was abandoned The comment in the code explicitly notes: "The old delta approach only returned rows for brand-new UserQuery events, leaving tool events within an ongoing session invisible. Full reload always shows current state." ## Column Structure ### activity-feed-hierarchical.html (main table, parent rows) thead columns: Event | Work Item | Time (3 columns: col-activity, work-item-cell, col-ts) NO status column in the thead. ### activity-feed-hierarchical-rows.html (delta rows, parent rows) td columns: col-activity | work-item-cell | col-status | col-ts (4 columns — HAS a col-status column with status-badge) ### activity-feed-children.html (expanded child rows, depth 1-3) td columns: col-activity | col-status | col-ts (3 columns — HAS col-status, but NO work-item-cell) ## CRITICAL BUG: Column Mismatch The main table thead has 3 columns (Event | Work Item | Time). But activity-feed-hierarchical-rows.html (delta rows) renders 4 tds (col-activity, work-item-cell, col-status, col-ts). And activity-feed-children.html renders 3 tds but they are (col-activity, col-status, col-ts) — missing work-item-cell, has status instead. This means STATUS column appears in expanded child rows but the thead does NOT have a STATUS column header. The child rows are misaligned with the parent table headers. ## Expanded State Save/Restore (dashboard-redesign.html) EXISTS in dashboard-redesign.html but has a key limitation: Save (htmx:beforeSwap): - Looks for .activity-row .expand-icon.expanded elements - Saves parent-row event_ids and "child:" + depth-1 event_ids - Stores in window._htmxFeedExpandedState Restore (htmx:afterSettle): - Calls handleParentClick(row) for each previously-expanded row - handleParentClick() triggers the HTMX lazy-load of children - This works but causes a SECOND async request per expanded row Problem: The save looks for .expand-icon.expanded — but the toggle function (toggleChildren) sets .expanded on the .expand-icon element. If the icon class name or structure ever diverges from what handleParentClick sets, the restore will silently fail. ## dashboard.html (WebSocket version) — No State Flicker dashboard.html does NOT have a flicker problem because: - It never does a full DOM replace - insertNewEventIntoActivityFeed() does surgical insertAdjacentHTML - Expanded state is preserved because the DOM is never destroyed - Trade-off: only shows new events via WS, does NOT update child counts etc. ## DOM Update Technique Summary | Dashboard | Mechanism | DOM Update | |-----------|-----------|------------| | dashboard.html | WebSocket /ws/events | Surgical insertAdjacentHTML | | dashboard-redesign.html | SSE /activity-feed/stream | Full htmx.ajax reload into #content-area | ## UI Techniques for Fixing the Flicker ### Option 1: morphdom / idiomorph (Virtual DOM diffing) - morphdom: https://github.com/patrick-steele-idem/morphdom Diffs existing DOM vs new HTML string, patches in-place. Preserves expanded state because elements are morphed, not replaced. Used by Phoenix LiveView. 230 bytes gzipped. - idiomorph: https://github.com/bigskysoftware/idiomorph Newer, handles id-based matching better. Used by HTMX 2.x swap extension. Key advantage: matches elements by id attribute, so rows stay stable. HTMX ships idiomorph as an extension: hx-ext="morph" + hx-swap="morph:outerHTML" ### Option 2: Save/Restore Expanded State (current approach, needs fixing) - Current implementation exists in dashboard-redesign.html but is brittle - Save: walk DOM, collect expanded event_ids before swap - Restore: after htmx:afterSettle, re-call handleParentClick per id - Problem: handleParentClick triggers a NEW HTMX fetch per expanded row (because hx-trigger="click once" — "once" means it only fires once per element; after full reload, the new elements have never been clicked, so they re-fetch) - This is actually correct behavior but causes N async requests on each refresh - Fix: ensure save uses data-event-id not element identity, restore is robust ### Option 3: HTMX morphdom Extension hx-swap="morph:innerHTML" with hx-ext="morph" - Load idiomorph extension script - Change hx-swap on the #content-area target to use morph - HTMX will diff the new HTML against existing DOM - Collapsed/expanded state preserved because class attributes on unchanged nodes are kept - Simplest fix: one script tag + change hx-swap value ### Option 4: CSS-only (not applicable) Cannot preserve expanded state across full DOM replacement with CSS alone. CSS state (like :checked on inputs) could work if we used hidden checkboxes for expand/collapse — but that would require a complete toggle mechanism rewrite. ### Option 5: Target tbody only, not #content-area Instead of replacing the entire #content-area (which includes the table, headers, filter bar, etc.), target only the tbody: htmx.ajax(GET, /views/activity-feed/delta?since=..., {target: #activity-feed-tbody, swap: afterbegin}) This prepends new rows without touching existing expanded rows. The abandoned delta endpoint exists for exactly this purpose. The reason it was abandoned (missing tool events in ongoing sessions) could be solved by also updating child counts on existing rows in a separate targeted request. ## Recommended Approach Option 3 (idiomorph/morph extension) is the cleanest fix: - Minimal code change (add script + change hx-swap) - Handles all edge cases (new rows, updated child counts, status changes) - No custom save/restore logic needed - HTMX officially supports this via hx-ext="morph" Option 2 (fix existing save/restore) is a lower-risk incremental fix: - The save/restore logic already exists in dashboard-redesign.html - Just needs to be made more robust - Works without adding new dependencies ## Files Involved - /Users/shakes/DevProjects/htmlgraph/src/python/htmlgraph/api/templates/dashboard-redesign.html Lines 241-333: refreshActivityFeed, SSE init, save/restore expand state - /Users/shakes/DevProjects/htmlgraph/src/python/htmlgraph/api/templates/partials/activity-feed-hierarchical.html Lines 1-110: Main table structure, thead (3 cols: Event | Work Item | Time) - /Users/shakes/DevProjects/htmlgraph/src/python/htmlgraph/api/templates/partials/activity-feed-hierarchical-rows.html Lines 1-77: Delta rows (4 cols including STATUS — MISMATCHED with thead) - /Users/shakes/DevProjects/htmlgraph/src/python/htmlgraph/api/templates/partials/activity-feed-children.html Lines 1-205: Child rows (3 cols: activity | STATUS | time — STATUS visible in child rows) - /Users/shakes/DevProjects/htmlgraph/src/python/htmlgraph/dashboard.html Lines 152-216: WebSocket connect/reconnect Lines 233-407: insertNewEventIntoActivityFeed (surgical DOM insertion, no flicker) Lines 903-913: toggleConversationTurn Lines 925-944: toggleChildEvent - /Users/shakes/DevProjects/htmlgraph/src/python/htmlgraph/api/routes/dashboard.py Lines 222-327: /views/activity-feed route (full render) Lines 514-632: /views/activity-feed/delta route (delta rows only) Lines 331-417: /views/activity-feed/children/{id} route (lazy child load)
