## Research Question What are industry-standard approaches for AI agent observability and distributed tracing, specifically for visualizing orchestrator + subagent activity trees in real-time with parent-child relationships? --- ## 1. OpenTelemetry (OTel) for AI Agent Tracing ### Status (2025) OTel is the undisputed industry standard for distributed tracing. The GenAI SIG (Special Interest Group) has been formalizing semantic conventions specifically for LLM and agent workloads throughout 2024-2025. ### Data Model OTel uses a Span hierarchy where every unit of work is a Span with: - trace_id (128-bit, globally unique per logical workflow) - span_id (64-bit, unique per span) - parent_span_id (links to parent, forming the tree) - start_time, end_time - attributes (key-value pairs, standardized via semantic conventions) - status (OK, ERROR, UNSET) - kind (CLIENT, SERVER, INTERNAL, PRODUCER, CONSUMER) ### GenAI Semantic Conventions (Official, 2025) **Agent Spans** (gen-ai-agent-spans spec): - gen_ai.operation.name = "invoke_agent" or "create_agent" - gen_ai.agent.id, gen_ai.agent.name, gen_ai.agent.version, gen_ai.agent.description - gen_ai.conversation.id (groups multi-turn conversations) - gen_ai.provider.name (e.g. "anthropic", "openai") - gen_ai.output.type - Span kind: CLIENT for cross-process agents, INTERNAL for same-process **LLM/Model Spans** (gen-ai-spans spec): - gen_ai.operation.name = "chat" or "generate_content" - gen_ai.provider.name, gen_ai.request.model - gen_ai.usage.input_tokens, gen_ai.usage.output_tokens - gen_ai.usage.cache_creation.input_tokens, gen_ai.usage.cache_read.input_tokens - gen_ai.input.messages, gen_ai.output.messages (opt-in, sensitive) - gen_ai.system_instructions, gen_ai.tool.definitions - Span name format: "{operation.name} {request.model}" e.g. "chat claude-3-5-sonnet" - Span kind: CLIENT **Tool Call Spans**: - gen_ai.operation.name = "execute_tool" - Are children of agent spans ### Parent-Child Relationships OTel solves the parent-child problem natively via context propagation. Each span carries a parent_span_id. For in-process code, context is passed via thread-local/contextvars. For cross-process (separate OS processes), context is propagated via W3C traceparent headers (or equivalent). ### Python SDK opentelemetry-sdk (official), plus: - openllmetry (Traceloop) - pip install traceloop-sdk, wraps 16+ LLM providers - openinference (Arize) - used by Phoenix - openlit - simpler init: openlit.init(otlp_endpoint=...) ### Can It Handle Async/Parallel Agents? YES. OTel uses Python contextvars under the hood, which correctly propagates context through asyncio. Each parallel Task() gets its own span that shares the same trace_id but has a distinct span_id with the orchestrator as parent. ### Overhead Minimal. OTel uses batch exporters (async, non-blocking). SDK adds ~1-5ms per span in worst case. For hook-based instrumentation (like HtmlGraph), this is irrelevant since hooks are already async. ### Key Gap for HtmlGraph OTel requires a running collector (Jaeger, Tempo, etc.) or a vendor backend. It is not designed for local SQLite-backed storage out of the box. This is HtmlGraph's differentiation opportunity. --- ## 2. AI-Specific Observability Tools ### LangSmith (LangChain, closed source) **Data Model:** - RunTree: hierarchical run objects, each with run_id, parent_run_id, run_type - run_type: "chain", "llm", "tool", "retriever", "embedding", "prompt" - @traceable decorator auto-detects nesting via contextvars - create_child() method for explicit hierarchy - Session = collection of root traces under one project **Strengths:** Automatic detection of nested calls, replay/comparison UI **Weaknesses:** Closed source, LangChain-centric, requires cloud ### Langfuse (Open Source, MIT) **Data Model (Traces > Observations):** - Trace: root container, has trace_id, session_id, user_id, metadata - Observation types: SPAN (generic), GENERATION (LLM call), EVENT (point-in-time) - Each observation has parent_observation_id for tree structure - Converging to OTel: natively supports OTel ingestion (frameworks like PydanticAI, smolagents emit OTel which Langfuse receives) - Agents visualized as graph to show complex agentic workflows **Strengths:** Truly open source, self-hostable, actively maintained (6M+ SDK installs/month), OTel-compatible, good multi-agent support **Weaknesses:** Requires Postgres + separate server, heavier infra than SQLite ### Arize Phoenix (Open Source, Apache 2.0) **Data Model:** - Pure OTel spans via openinference instrumentation library - Spans stored in PostgreSQL or in-memory SQLite for local use - UI shows trace waterfall view (tree of spans with timing bars) - Supports: OpenAI Agents SDK, LangGraph, CrewAI, LlamaIndex, DSPy **Strengths:** Local/offline mode with SQLite, pure OTel data model, no vendor lock-in **Weaknesses:** Evaluation-focused, less suited for real-time live dashboard ### Key Pattern Across All Tools Every mature observability tool converges on the same core model: trace_id (groups the whole workflow) + span_id / parent_span_id (tree structure) + timestamp + duration (timing) + attributes (metadata: model, tokens, agent name, tool name) --- ## 3. W3C Trace Context (traceparent propagation) ### Format traceparent: 00-{trace_id_32hex}-{parent_id_16hex}-{flags_2hex} Example: "00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01" Fields: - version: 2 chars (always "00" currently) - trace-id: 32 hex chars (128-bit, stays constant for entire trace) - parent-id: 16 hex chars (64-bit, changes at each service/process boundary) - trace-flags: 2 hex chars ("01" = sampled, "00" = not sampled) ### Propagation Between OS Processes For HTTP: via HTTP headers (standard) For async jobs/message queues: embed traceparent in message payload For subprocess / OS process spawning: can pass via environment variables **Key insight for HtmlGraph:** Claude Code spawns subagents as separate OS processes (Claude Code task tool). The orchestrator session_id / trace context could be passed to subagents via environment variables set before subprocess launch. Each subagent hook would read TRACEPARENT env var to establish parent linkage. ### Current HtmlGraph Approach vs W3C HtmlGraph currently uses parent_session_id + parent_event_id stored in SQLite for parent-child linking. This is functionally equivalent to trace-id + parent-id but uses UUIDs and DB FKs instead of the 128/64-bit OTel format. The concept is identical; the encoding differs. --- ## 4. SQLite WAL Mode for Real-Time Reads ### Concurrency Model (from official SQLite docs) WAL (Write-Ahead Log) mode allows: - Unlimited concurrent readers (no reader blocks another reader) - Readers do NOT block writers - Writers do NOT block readers - Only one writer at a time (serialized writes) ### Performance for 100ms Polling Dashboard SUITABLE with caveats: - Read query at 100ms intervals: fine, no blocking - WAL file grows if checkpoints cannot complete - Checkpoint starvation: if there is ALWAYS an active reader (e.g. persistent connection), the WAL file never gets checkpointed and grows without bound, degrading read performance proportionally to WAL file size - Solution: use short-lived read connections (open, query, close) rather than persistent connections; or explicitly call PRAGMA wal_checkpoint(PASSIVE) periodically ### Recommendation for HtmlGraph Dashboard - Enable WAL: PRAGMA journal_mode=WAL (set once on DB init) - Use short-lived read connections for dashboard polling - Set PRAGMA wal_autocheckpoint=1000 (checkpoint every 1000 pages) - For SSE streaming endpoint: open new connection per poll, do not hold persistent read connection - At 100ms polling with ~100 events/session, WAL mode is well within safe operating range --- ## 5. SSE vs WebSocket for Live Activity Dashboard ### Server-Sent Events (SSE) - Unidirectional: server → client only - Uses standard HTTP (works through proxies, firewalls, HTTP/2 multiplexing) - Browser auto-reconnects with exponential backoff - htmx SSE extension: hx-ext="sse" + sse-connect="/stream" + sse-swap="event-name" - Perfect for: live feeds, dashboards, log streaming, activity trees - Failure modes: connection drops are handled automatically; no binary support; no client-to-server streaming ### WebSocket - Bidirectional: full duplex - Requires connection upgrade, can be blocked by some proxies - No automatic reconnect (must implement manually) - Better for: interactive apps (chat, collaborative editing, gaming) - Worse for: read-heavy dashboards ### Verdict for HtmlGraph Activity Tree SSE is the right choice. The live activity feed is server-push only: - New tool call events appear - Agent statuses update (running → complete) - Timing bars update The only client actions (filtering, expanding nodes) are occasional and can use normal HTTP requests (htmx hx-get). SSE + htmx is a well-proven pattern for exactly this use case. ### htmx SSE Extension Specifics - Install: hx-ext="sse" on container element - Connect: sse-connect="/api/live-events" - Swap: sse-swap="tool_call" hx-swap="beforeend" hx-target="#event-list" - Auto-reconnect: built-in exponential backoff - Limitation: no catch-all handler, must name each event type explicitly --- ## HtmlGraph Current State vs Industry Standards ### What HtmlGraph Already Has (Aligned with Industry) 1. parent_event_id + parent_session_id in agent_events table = OTel's parent_span_id + trace_id 2. event_type (tool_call, tool_result, etc.) = OTel's span kinds 3. session_id as trace grouping = OTel's trace_id concept 4. tool_traces table for PreToolUse/PostToolUse correlation = OTel's span start/end timing 5. claude_task_id for subagent grouping = OTel's operation context ### What HtmlGraph Is Missing (Industry Gap) 1. No WAL mode enabled on SQLite (checkpoint starvation risk with live dashboard polling) 2. No W3C traceparent propagation between sessions (parent-child links are established via DB lookup, not via context injection at spawn time — this is why linkage is fragile) 3. No standardized span names ("invoke_agent {agent_name}", "chat {model}", "execute_tool {tool_name}") 4. No duration tracking on events (start_time + end_time → execution_duration_seconds exists but is not always populated) 5. No live SSE endpoint (dashboard currently polls; WAL + SSE would enable true push) 6. Span/event type taxonomy is ad-hoc vs OTel's standardized gen_ai.operation.name values ### The Core Attribution Problem (and the Solution) The hardest problem in multi-agent tracing: when Claude Code spawns a subagent via Task(), the subagent runs in a new OS process with no shared memory. How does the subagent know who its parent is? Industry solution (OTel): inject traceparent into the spawned process via: - Environment variables (TRACEPARENT=00-{trace_id}-{parent_span_id}-01) - Message payload (for queue-based systems) HtmlGraph current approach: DB lookup (find most recent UserQuery, infer parent). This works but is fragile — it can misattribute events if two sessions are active simultaneously. Better approach: At Task() spawn time, write a context file or set an env var containing the orchestrator session_id and the spawning event_id. The subagent's session-start hook reads this to establish explicit parent linkage without DB inference. --- ## Recommendations ### Immediate Wins (Low Effort) 1. Enable SQLite WAL mode in HtmlGraphDB.__init__: PRAGMA journal_mode=WAL 2. Add PRAGMA wal_autocheckpoint=1000 to prevent WAL file bloat 3. Add SSE endpoint to dashboard server for live event streaming (replaces 100ms polling with push) ### Medium Term (Adopt Industry Standard Concepts) 4. Standardize event taxonomy to align with OTel GenAI conventions: - "invoke_agent {agent_name}" for Task() delegations - "chat {model}" for LLM completions - "execute_tool {tool_name}" for individual tool calls 5. Add explicit trace_id field to sessions table (= current session_id, just rename conceptually) 6. Add explicit span_id to agent_events (= current event_id, just make it explicit) 7. Implement TRACEPARENT env var injection at Task() spawn time for reliable parent-child attribution ### Longer Term (OTel Compatibility) 8. Add OTel OTLP export endpoint so HtmlGraph events can be forwarded to Jaeger/Tempo 9. Accept OTel OTLP ingestion so external frameworks (LangGraph, CrewAI) can send spans to HtmlGraph 10. This makes HtmlGraph a local-first OTel collector with HTML/SQLite storage — a unique niche ### What to Build From Scratch (HtmlGraph's Unique Value) - HTML file storage (not OTel standard, but human-readable and offline-first) - Graph traversal via CSS selectors (unique) - Live tree visualization in browser with no external backend - Claude Code-specific hook integration (PreToolUse/PostToolUse correlation) - The SQLite + WAL + SSE stack is simpler than any OTel collector setup ### What to Adopt From Industry - OTel span vocabulary and attribute names for the fields that exist - W3C traceparent format for cross-process context propagation - SSE for live push (already proven pattern in htmx ecosystem) - Langfuse's Trace > Observation > Generation hierarchy as mental model for the dashboard UI --- ## Sources - OTel AI Agent Observability Blog: https://opentelemetry.io/blog/2025/ai-agent-observability/ - OTel GenAI Agent Spans Spec: https://opentelemetry.io/docs/specs/semconv/gen-ai/gen-ai-agent-spans/ - OTel GenAI Model Spans Spec: https://opentelemetry.io/docs/specs/semconv/gen-ai/gen-ai-spans/ - W3C Trace Context: https://www.w3.org/TR/trace-context/ - W3C Traceparent Explained: https://www.dash0.com/knowledge/w3c-trace-context-traceparent-tracestate - SQLite WAL Mode: https://sqlite.org/wal.html - Langfuse Architecture: https://langfuse.com/blog/2024-07-ai-agent-observability-with-langfuse - Arize Phoenix: https://github.com/Arize-ai/phoenix - OpenLLMetry: https://github.com/traceloop/openllmetry - htmx SSE Extension: https://v1.htmx.org/extensions/server-sent-events/ - LangSmith Deep Dive: https://medium.com/@aviadr1/langsmith-tracing-deep-dive-beyond-the-docs-75016c91f747
