## Problem When orchestrator launches subagents via Task()/Agent(), the orchestrator's own subsequent tool calls (Bash, browser_wait_for, etc.) get incorrectly nested UNDER the Task node in the dashboard activity feed. They should be siblings of the Task node, not children. ## Visual Evidence Dashboard shows Task htmlgraph:researcher (6) expanded with: - mcp__plugin_playwright browser_wait_for — THIS IS ORCHESTRATOR'S CALL (wrong parent) - Bash "Check attribution" — THIS IS ORCHESTRATOR'S CALL (wrong parent) - Grep, Read, Agent — These ARE the researcher's calls (correct nesting) ## Root Cause pretooluse.py assigns parent_event_id by querying most recent task_delegation row. When orchestrator makes tool calls AFTER spawning a Task(), the hook finds that Task's delegation record and incorrectly assigns it as parent. Cannot distinguish orchestrator direct calls from subagent calls because: - Claude Code does NOT send agent_id in PreToolUse hook_input - Both orchestrator and subagent share the same session_id - No way to tell "this Bash call is from the orchestrator" vs "this Bash call is from the subagent" ## Key Files - packages/claude-plugin/hooks/scripts/pretooluse-integrator.py (PreToolUse hook entry) - src/python/htmlgraph/hooks/pretooluse.py (assigns parent_event_id, look for task_delegation query) - src/python/htmlgraph/hooks/event_tracker.py (tracks task_delegation records) ## Suggested Fix Option A: Timing — if Task has a SubagentStop event (completed), subsequent calls should NOT be children Option B: Check if Task is still running — only nest if Task hasn't completed yet Option C: Use agent_id from event — if it matches orchestrator agent, don't nest under Task ## Verified v0.33.57, 2026-03-10
