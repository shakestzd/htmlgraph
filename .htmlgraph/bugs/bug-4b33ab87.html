Description

When the orchestrator delegates a trivial file-read task to htmlgraph:researcher, the agent loads the agent-context skill (and any other frontmatter-injected skills) as part of its startup sequence. The skill injection dominates wall-clock time for what should be a <100ms cat operation — observed today with a read of .htmlgraph/plans/plan-368a5cd2.yaml which stalled long enough that the user complained 'what is taking so long'. Inspecting the async output file showed the agent had only just finished ingesting its skill context, not yet begun the read. Two problems: (1) skill injection cost is not proportional to task complexity — a simple read pays the same overhead as a deep research task; (2) orchestrator delegation rules push ALL file reads through researcher, so the overhead is incurred on every Read tool call. Possible mitigations: (a) carve out a lightweight reader agent variant without skill injection for trivial reads, (b) make skill injection lazy/on-demand rather than eager at agent startup, (c) relax orchestrator rules to permit direct Read tool for data files (YAML, JSON, small config) while still delegating source-code reads and edits, (d) cache skill context between agent invocations so only the first read pays the cost.
