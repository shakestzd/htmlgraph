Description

wipnote serve runs a background indexer (~5s loop) issuing begin-immediate write txns to ingest sessions, plus a separate writequeue. Under DELETE-journal single-writer semantics (correct for overlayfs) the indexer contends with its own writequeue and periodic CLI opens, producing a self-perpetuating SQLITE_BUSY storm: every begin-immediate fails database is locked (5), the orphan filter query fails, the read-index never updates, and dashboard data endpoints degrade to unavailable right now (metric tiles show --). Even plain wipnote session list and wipnote bug create fail at applying PRAGMA journal_mode / read user_version. The running serve already contains deployed fixes bug-74a7bda7 / bug-7dbaf552 / bug-dd5db2d1 (binary 0.59.1-582-gc8646013c, not replaced under the process) so those RetryOnBusy fixes are INSUFFICIENT for the serve background write loop, a path they did not cover. Likely fix: route all serve-side writes through the single writequeue (stop the indexer issuing independent begin-immediate), add RetryOnBusy + exponential backoff to the indexer path, and back off hard when busy instead of hammering every 5s. Reproduced live 2026-05-18 with no external concurrent launcher processes running, so this is serve self-contention not external load.
