Description

On a fresh git clone the canonical .wipnote/*.html files contain lineage edges, but 'wipnote lineage ' returns 'no related nodes' because the SQLite read-index is per-user/derived and not committed, and is not built on first run. A brand-new OSS user running the headline lineage command cold gets an empty result -> bounces. Launch-blocking for the OSS Show-HN flag-plant. Repro: clone repo to a fresh dir with CLAUDE_PROJECT_DIR unset, run 'wipnote lineage' on a committed feature id. Fix direction: lazily build/refresh the read index on first read when missing/stale (must NOT depend on a hazardous background reindex), or ship a documented one-line bootstrap that the README quickstart runs before the aha. Discovered during feat-5d2188b9 cold-clone audit.
