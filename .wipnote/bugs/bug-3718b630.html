Description

When a work item's code is written by a delegated subagent, the orchestrator's 'wipnote complete ' is double-gated: (a) it needs a session-scoped passing 'wipnote check --gate' run by the orchestrator itself, because a subagent's gate record is session-bound and does not count; and (b) it requires --accepted-advisory because subagent commits are not auto-linked to the active work item. This is a still-live attribution gap in the same family as bug-63f42308 (which is done and scoped only to Codex hook-event feature_id population). The fix must land in the binary/hooks — auto-link subagent commits to the active work item and/or make gate records visible to the completing orchestrator session — NOT be papered over by per-user memory or advisory skill prose. Memory/advisory enforcement creates a false positive: completion appears to work via a memorized workaround while the underlying defect stays hidden, masking the real bug from all wipnote users.
