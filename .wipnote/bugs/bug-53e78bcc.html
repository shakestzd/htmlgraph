Description

share_paths.go isValidHarnessTree checks for treeRoot/.agents/plugins/marketplace.json as the codex-marketplace sentinel. The repo source has that exact path under packages/codex-marketplace/, so dev mode works. But the GoReleaser archive flattens .agents/plugins/ out — the published tarball has codex-marketplace/marketplace.json at the root with no .agents/plugins/ prefix. Result: every binary-install user gets 'wipnote codex-marketplace tree not found' even when the tree is correctly installed. WIPNOTE_CODEX_DIR override does NOT bypass validation, so the error is unavoidable from a binary install. Same risk pattern likely affects wipnote gemini (different sentinel — gemini-extension.json — needs separate verification). wipnote claude is unaffected (its sentinel .claude-plugin/plugin.json survives the flattening). Fix requires choosing one of: (a) update validator to accept the flattened layout, (b) update GoReleaser/plugin-build-ports to preserve deep layout, (c) consolidate so dev source and bundled tree share one layout. Out-of-scope for tonight's launch flag-plant since codex/gemini are documented experimental; needs focused follow-up. NOT a blocker for wipnote claude dogfooding or the Show-HN positioning, which is the Claude path.
